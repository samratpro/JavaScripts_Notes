## 1. What Are We Learning?

**Electron.js** is a framework for building native-feeling desktop applications using web technologies. Apps you already use — VS Code, Slack, Discord, WhatsApp Desktop, Figma Desktop — are built with Electron.

**React** is a JavaScript library for building user interfaces from reusable components.

**TypeScript** is a superset of JavaScript that adds static types. It catches bugs at compile time and gives students great auto-complete in VS Code.

When we combine all three:

- **React** handles the user interface inside the app window.
- **Electron** wraps that UI in a native desktop window and gives it access to the operating system.
- **TypeScript** makes the IPC bridge between them type-safe, so students can't accidentally send the wrong data.

**End result:** A single, type-safe codebase that runs as a standalone app on Windows, macOS, and Linux.

## 3. How Electron Works (Mental Model)

Every Electron app has **two types of processes**:

| Process | What It Is | Analogy |
|---|---|---|
| **Main process** | A Node.js program. Controls windows, menus, OS-level features. One per app. | The "manager" |
| **Renderer process** | A Chromium browser tab that renders your UI (your React app lives here). | The "worker/display" |

They run in **separate memory spaces** and cannot directly share variables. They talk to each other through a system called **IPC (Inter-Process Communication)**.

```
┌────────────────────────────────────────────┐
│               MAIN PROCESS                 │
│   (Node.js + TS — full OS access)          │
│   - Creates windows                        │
│   - Reads/writes files                     │
│   - Shows native dialogs                   │
└────────────┬───────────────────────────────┘
             │ typed IPC messages
             ▼
┌────────────────────────────────────────────┐
│            RENDERER PROCESS                │
│   (Chromium + React + TS — sandboxed)      │
│   - Your React app runs here               │
│   - HTML / CSS / TSX UI                    │
└────────────────────────────────────────────┘
```

**Golden rule:** The main process has full OS power. The renderer should be treated like any browser page — it should *not* get raw OS access. Communication happens through a secure bridge called the **preload script**, and TypeScript makes that bridge type-safe.

## 5. Lesson 1: Your First Electron App (TypeScript)

**Goal:** Open a native window that displays an HTML page — written entirely in TypeScript.

### Step 1 — Create `electron/main.ts` (the main process)

```typescript
// electron/main.ts
import { app, BrowserWindow } from 'electron';
import * as path from 'path';

function createWindow(): void {
  const win = new BrowserWindow({
    width: 900,
    height: 650,
    webPreferences: {
      contextIsolation: true,
      nodeIntegration: false,
    },
  });

  // Load an HTML file into the window
  win.loadFile(path.join(__dirname, '../index.html'));
}

// Wait until Electron is ready, then open a window
app.whenReady().then(createWindow);

// Quit the app when all windows are closed (except on macOS)
app.on('window-all-closed', (): void => {
  if (process.platform !== 'darwin') app.quit();
});

// On macOS, re-open a window when the dock icon is clicked
app.on('activate', (): void => {
  if (BrowserWindow.getAllWindows().length === 0) createWindow();
});
```

### Step 2 — Create `index.html`

```html
<!DOCTYPE html>
<html>
  <head>
    <meta charset="UTF-8" />
    <title>Hello Electron + TypeScript</title>
  </head>
  <body>
    <h1>Hello, Electron!</h1>
    <p>This window is powered by TypeScript.</p>
  </body>
</html>
```

### Step 3 — Update `package.json`

```json
{
  "name": "hello-electron-react-ts",
  "version": "1.0.0",
  "main": "dist-electron/main.js",
  "scripts": {
    "build:electron": "tsc -p tsconfig.json",
    "start": "npm run build:electron && electron ."
  },
  "devDependencies": {
    "electron": "^30.0.0",
    "typescript": "^5.4.0",
    "@types/node": "^20.0.0"
  }
}
```

Note that `main` points to the **compiled** JavaScript in `dist-electron/`, not the TypeScript source.

### Step 4 — Run it

```bash
npm start
```

TypeScript compiles `electron/main.ts` into `dist-electron/main.js`, then Electron runs it. A native window opens.

### TypeScript quick intro

If students haven't seen TypeScript, here's the minimum they need:


## 6. Lesson 2: Adding React with TypeScript

So far we rendered plain HTML. Let's replace that with a React app written in TypeScript.

### Option A — Quick start with Vite + React + TS (recommended)

```bash
npm create vite@latest renderer -- --template react-ts
cd renderer
npm install
npm run build
cd ..
```

The `react-ts` template sets up React with TypeScript automatically. Vite outputs static files into `renderer/dist/`.

Update `electron/main.ts` to load the built React page:

```typescript
win.loadFile(path.join(__dirname, '../renderer/dist/index.html'));
```

### Option B — Development mode with hot reload

During development, Vite runs a dev server (e.g. `http://localhost:5173`). Update `electron/main.ts` to load from the dev server in development mode:

```typescript
const isDev: boolean = !app.isPackaged;

if (isDev) {
  win.loadURL('http://localhost:5173');
} else {
  win.loadFile(path.join(__dirname, '../renderer/dist/index.html'));
}
```

Add convenient scripts to the root `package.json`:

```json
{
  "scripts": {
    "dev:renderer": "cd renderer && npm run dev",
    "dev:electron": "npm run build:electron && electron .",
    "build:electron": "tsc -p tsconfig.json",
    "build:renderer": "cd renderer && npm run build",
    "start": "npm run build:renderer && npm run build:electron && electron ."
  }
}
```

In one terminal run `npm run dev:renderer`, in another run `npm run dev:electron`. You now have hot-reloading React + TypeScript inside Electron.

## 7. Lesson 3: React + TypeScript Fundamentals

Before building more, let's cover the React + TS concepts students need.

### 7.1 Components with typed props

A **component** is a function that returns JSX. With TypeScript, we describe its props using an `interface` or `type`.

```tsx
// Greeting.tsx
interface GreetingProps {
  name: string;
}

function Greeting({ name }: GreetingProps) {
  return <h1>Hello, {name}!</h1>;
}

export default Greeting;
```

### 7.2 Props with optional fields and defaults

```tsx
interface ButtonProps {
  label: string;
  disabled?: boolean;        // optional
  onClick: () => void;       // function prop
}

function Button({ label, disabled = false, onClick }: ButtonProps) {
  return (
    <button disabled={disabled} onClick={onClick}>
      {label}
    </button>
  );
}
```

### 7.3 State with `useState`

TypeScript can usually infer the state type from its initial value. For complex types, pass a generic.

```tsx
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState<number>(0);

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>Click me</button>
    </div>
  );
}
```

### 7.4 Side effects with `useEffect`

```tsx
import { useEffect, useState } from 'react';

function Clock() {
  const [time, setTime] = useState<Date>(new Date());

  useEffect(() => {
    const id: NodeJS.Timeout = setInterval(() => setTime(new Date()), 1000);
    return () => clearInterval(id);
  }, []);

  return <p>The time is {time.toLocaleTimeString()}</p>;
}
```

### 7.5 Lists and keys (with a typed array)

```tsx
interface TodoItem {
  id: number;
  text: string;
  done: boolean;
}

interface TodoListProps {
  items: TodoItem[];
}

function TodoList({ items }: TodoListProps) {
  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>
          {item.done ? '[x]' : '[ ]'} {item.text}
        </li>
      ))}
    </ul>
  );
}
```

### 7.6 Forms and typed event handlers

React provides types like `ChangeEvent<HTMLInputElement>` and `FormEvent<HTMLFormElement>`.

```tsx
import { useState, ChangeEvent, FormEvent } from 'react';

function NameForm() {
  const [name, setName] = useState<string>('');

  function handleChange(e: ChangeEvent<HTMLInputElement>): void {
    setName(e.target.value);
  }

  function handleSubmit(e: FormEvent<HTMLFormElement>): void {
    e.preventDefault();
    alert(`Hello, ${name}`);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input value={name} onChange={handleChange} placeholder="Your name" />
      <button type="submit">Greet</button>
    </form>
  );
}
```

### 7.7 Conditional rendering with union types

```tsx
interface User {
  name: string;
}

interface LoginProps {
  user: User | null;
}

function Login({ user }: LoginProps) {
  if (!user) return <p>Please log in</p>;
  return <p>Welcome back, {user.name}</p>;
}
```

## 8. Lesson 4: Main ↔ Renderer Communication (IPC)

React (renderer) cannot directly read files or show OS dialogs. It must ask the main process. Electron provides two modules:

- `ipcMain` — used in `main.ts`
- `ipcRenderer` — used in the renderer (exposed safely via the preload script)

### Patterns

| Pattern | Use case |
|---|---|
| `ipcRenderer.send` / `ipcMain.on` | Fire-and-forget (one-way) |
| `ipcRenderer.invoke` / `ipcMain.handle` | Request/response (recommended) |
| `webContents.send` / `ipcRenderer.on` | Main → Renderer push |

We'll use `invoke`/`handle` because it's promise-based and plays very well with TypeScript.

### Shared IPC type contract

Create a shared type file that both sides import:

```typescript
// electron/types.ts
export interface IElectronAPI {
  getAppVersion: () => Promise<string>;
  readFile: (path: string) => Promise<string>;
  saveFile: (path: string, data: string) => Promise<boolean>;
  openFile: () => Promise<string | null>;
  saveNote: (content: string) => Promise<boolean>;
  loadNote: () => Promise<string>;
}
```

## 9. Lesson 5: Preload Script, Context Isolation, and Typed APIs

The preload script runs **before** the web page loads. It exposes a safe, typed API to the renderer through `contextBridge`.

### Step 1 — Create `electron/preload.ts`

```typescript
// electron/preload.ts
import { contextBridge, ipcRenderer } from 'electron';
import type { IElectronAPI } from './types';

const api: IElectronAPI = {
  getAppVersion: () => ipcRenderer.invoke('app:getVersion'),
  readFile: (path: string) => ipcRenderer.invoke('fs:readFile', path),
  saveFile: (path: string, data: string) =>
    ipcRenderer.invoke('fs:saveFile', { path, data }),
  openFile: () => ipcRenderer.invoke('dialog:openFile'),
  saveNote: (content: string) => ipcRenderer.invoke('note:save', content),
  loadNote: () => ipcRenderer.invoke('note:load'),
};

contextBridge.exposeInMainWorld('api', api);
```

### Step 2 — Declare `window.api` in the renderer

Create `renderer/src/types/global.d.ts` so TypeScript knows `window.api` exists:

```typescript
// renderer/src/types/global.d.ts
import type { IElectronAPI } from '../../../electron/types';

declare global {
  interface Window {
    api: IElectronAPI;
  }
}

export {};
```

Now `window.api.getAppVersion()` is fully typed in every `.tsx` file.

### Step 3 — Wire up the preload in `main.ts`

```typescript
const win = new BrowserWindow({
  width: 900,
  height: 650,
  webPreferences: {
    preload: path.join(__dirname, 'preload.js'),   // compiled .js, not .ts
    contextIsolation: true,
    nodeIntegration: false,
  },
});
```

### Step 4 — Handle requests in the main process

```typescript
// electron/main.ts
import { app, ipcMain, IpcMainInvokeEvent } from 'electron';
import * as fs from 'fs/promises';

ipcMain.handle('app:getVersion', (): string => app.getVersion());

ipcMain.handle(
  'fs:readFile',
  async (event: IpcMainInvokeEvent, filePath: string): Promise<string> => {
    return await fs.readFile(filePath, 'utf-8');
  }
);

interface SaveFilePayload {
  path: string;
  data: string;
}

ipcMain.handle(
  'fs:saveFile',
  async (event: IpcMainInvokeEvent, { path: filePath, data }: SaveFilePayload): Promise<boolean> => {
    await fs.writeFile(filePath, data, 'utf-8');
    return true;
  }
);
```

### Step 5 — Use it from React with full type safety

```tsx
import { useEffect, useState } from 'react';

function AppVersion() {
  const [version, setVersion] = useState<string>('...');

  useEffect(() => {
    window.api.getAppVersion().then(setVersion);
  }, []);

  return <p>App version: {version}</p>;
}
```

If a student mistypes `window.api.getAppVerion()`, TypeScript catches it immediately.

## 10. Lesson 6: File System, Dialogs, Menus

### 10.1 Native "Open File" dialog

```typescript
// electron/main.ts
import { dialog } from 'electron';

ipcMain.handle('dialog:openFile', async (): Promise<string | null> => {
  const result = await dialog.showOpenDialog({
    properties: ['openFile'],
    filters: [{ name: 'Text', extensions: ['txt', 'md'] }],
  });
  if (result.canceled) return null;
  return result.filePaths[0];
});
```

Use it in React (already typed via `global.d.ts`):

```tsx
async function handleOpen(): Promise<void> {
  const path: string | null = await window.api.openFile();

}
```

```typescript
// electron/main.ts
import { Notification } from 'electron';

interface NotifyPayload {

ipcMain.handle(

);
```

## 11. Lesson 7: Building a Real App — Notes Taker

Let's put everything together in a small, fully-typed app that lets the user write notes and save them to disk.

### `electron/types.ts` (shared contract)

```typescript
export interface IElectronAPI {
  saveNote: (content: string) => Promise<boolean>;
  loadNote: () => Promise<string>;
}
```

### `electron/preload.ts`

```typescript
import { contextBridge, ipcRenderer } from 'electron';
import type { IElectronAPI } from './types';

const api: IElectronAPI = {
  saveNote: (content: string) => ipcRenderer.invoke('note:save', content),
  loadNote: () => ipcRenderer.invoke('note:load'),
};

contextBridge.exposeInMainWorld('api', api);
```

### `electron/main.ts` (handlers)

```typescript
import { app, ipcMain, dialog, IpcMainInvokeEvent } from 'electron';
import * as fs from 'fs/promises';

ipcMain.handle(
  'note:save',
  async (event: IpcMainInvokeEvent, content: string): Promise<boolean> => {
    const { canceled, filePath } = await dialog.showSaveDialog({
      defaultPath: 'note.txt',
      filters: [{ name: 'Text', extensions: ['txt'] }],
    });
    if (canceled || !filePath) return false;
    await fs.writeFile(filePath, content, 'utf-8');
    return true;
  }
);

ipcMain.handle('note:load', async (): Promise<string> => {
  const { canceled, filePaths } = await dialog.showOpenDialog({
    properties: ['openFile'],
    filters: [{ name: 'Text', extensions: ['txt', 'md'] }],
  });
  if (canceled || filePaths.length === 0) return '';
  return await fs.readFile(filePaths[0], 'utf-8');
});
```

### React component — `Notes.tsx`

```tsx
import { useState, ChangeEvent } from 'react';

export default function Notes() {
  const [text, setText] = useState<string>('');
  const [status, setStatus] = useState<string>('');

  async function handleSave(): Promise<void> {
    const ok: boolean = await window.api.saveNote(text);
    setStatus(ok ? 'Saved!' : 'Save cancelled');
  }

  async function handleLoad(): Promise<void> {
    const loaded: string = await window.api.loadNote();
    setText(loaded);
    setStatus('Loaded');
  }

  function handleChange(e: ChangeEvent<HTMLTextAreaElement>): void {
    setText(e.target.value);
  }

  return (
    <div style={{ padding: 20, fontFamily: 'sans-serif' }}>
      <h1>My Notes</h1>
      <textarea
        value={text}
        onChange={handleChange}
        rows={15}
        cols={60}
        placeholder="Type your notes here..."
      />
      <div style={{ marginTop: 10 }}>
        <button onClick={handleSave}>Save</button>
        <button onClick={handleLoad} style={{ marginLeft: 10 }}>Load</button>
        <span style={{ marginLeft: 20, color: 'green' }}>{status}</span>
      </div>
    </div>
  );
}
```

### `App.tsx`

```tsx
import Notes from './components/Notes';

export default function App() {
  return <Notes />;
}
```

### `main.tsx`

```tsx
import React from 'react';
import ReactDOM from 'react-dom/client';
import App from './App';
import './styles/app.css';

ReactDOM.createRoot(document.getElementById('root') as HTMLElement).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);
```

Run the app and students have a working, saveable, loadable notes editor — with full type safety on both sides of the IPC boundary.

## 12. Lesson 8: Packaging and Distribution

Shipping an Electron app has three stages:

1. **Compile** TypeScript → JavaScript and bundle React with Vite.
2. **Package** the app into a distributable folder or installer.
3. **Distribute** the installer to users.

There are two common tools for stage 2 on Windows:

| Tool | What it produces | When to use |
|---|---|---|
| **electron-builder** | `.exe`, `.msi`, `.dmg`, `.AppImage`, `.deb`, etc. All-in-one. | Most apps. Recommended. |
| **electron-packager + Inno Setup** | Unpacked app folder + custom `.exe` installer you fully control | You want a branded, customized Windows installer |

We'll cover both.

### 12.1 Option A — All-in-one with electron-builder

```bash
npm install --save-dev electron-builder
```

Add to `package.json`:

```json
{
  "build": {
    "appId": "com.example.notesapp",
    "productName": "Notes App",
    "copyright": "Copyright © 2026 Your Name",
    "files": [
      "dist-electron/**/*",
      "renderer/dist/**/*"
    ],
    "directories": {
      "output": "release"
    },
    "mac": { "category": "public.app-category.productivity" },
    "win": {
      "target": "nsis",
      "icon": "assets/icon.ico"
    },
    "nsis": {
      "oneClick": false,
      "allowToChangeInstallationDirectory": true,
      "createDesktopShortcut": true,
      "createStartMenuShortcut": true,
      "shortcutName": "Notes App"
    },
    "linux": { "target": "AppImage" }
  },
  "scripts": {
    "dist": "npm run build:renderer && npm run build:electron && electron-builder"
  }
}
```

Build installers:

```bash
npm run dist
```

You'll find `Notes App Setup 1.0.0.exe` in the `release/` folder. That's a working Windows installer, ready to send to users.

### 12.2 Option B — Unpacked build + Inno Setup

This route gives you full control over the installer UI — custom license pages, optional components, custom install directory logic, branded graphics, etc. It's how many professional Windows apps (including older versions of VS Code) were shipped.

The pipeline is:

```
React + TS source  →  Vite + tsc compile  →  electron-packager (unpacked folder)  →  Inno Setup (.iss)  →  branded .exe installer
```

#### Step 1 — Install electron-packager

`electron-packager` produces an **unpacked** application folder: your app bundled with the Electron runtime, but without an installer wrapped around it.

```bash
npm install --save-dev @electron/packager
```

Add a script to `package.json`:

```json
{
  "scripts": {
    "build:renderer": "cd renderer && npm run build",
    "build:electron": "tsc -p tsconfig.json",
    "build": "npm run build:renderer && npm run build:electron",
    "pack:win": "npm run build && electron-packager . NotesApp --platform=win32 --arch=x64 --out=release --overwrite --icon=assets/icon.ico --ignore=\"^/(renderer/src|renderer/node_modules|electron)$\""
  }
}
```

Run it:

```bash
npm run pack:win
```

You'll get a folder like this:

```
release/
└── NotesApp-win32-x64/
    ├── NotesApp.exe            ← runnable, but not installable yet
    ├── resources/
    │   └── app/                ← your compiled app
    ├── locales/
    ├── *.dll
    └── LICENSE.electron.txt
```

Double-clicking `NotesApp.exe` launches the app. But users can't install it with shortcuts, registry entries, etc. That's what Inno Setup is for.

#### Step 2 — Install Inno Setup

Inno Setup is a free Windows installer builder. Download and install it:

- Website: [https://jrsoftware.org/isinfo.php](https://jrsoftware.org/isinfo.php)
- Install the latest version (currently Inno Setup 6.x)
- It includes an IDE called **Inno Setup Compiler** and a command-line tool **`ISCC.exe`**

After install, verify `ISCC.exe` is available:

```bash
"C:\Program Files (x86)\Inno Setup 6\ISCC.exe" /?
```

#### Step 3 — Create `installer/NotesApp.iss`

Create a folder `installer/` at the project root and a file called `NotesApp.iss`:

```iss
; NotesApp.iss — Inno Setup script for Notes App
; Docs: https://jrsoftware.org/ishelp/

#define MyAppName "Notes App"
#define MyAppVersion "1.0.0"
#define MyAppPublisher "Your Name"
#define MyAppURL "https://example.com"
#define MyAppExeName "NotesApp.exe"
#define SourceDir "..\release\NotesApp-win32-x64"

[Setup]
; A unique GUID identifies this app to Windows. Generate a new one in Inno Setup
; Compiler via Tools -> Generate GUID. Never reuse between different apps.
AppId={{B9C4F2D7-3E1A-4C52-A9F8-0F7C2E8B1A44}
AppName={#MyAppName}
AppVersion={#MyAppVersion}
AppPublisher={#MyAppPublisher}
AppPublisherURL={#MyAppURL}
AppSupportURL={#MyAppURL}
AppUpdatesURL={#MyAppURL}
DefaultDirName={autopf}\{#MyAppName}
DefaultGroupName={#MyAppName}
DisableProgramGroupPage=yes
OutputDir=..\release\installer
OutputBaseFilename=NotesApp-Setup-{#MyAppVersion}
SetupIconFile=..\assets\icon.ico
UninstallDisplayIcon={app}\{#MyAppExeName}
Compression=lzma
SolidCompression=yes
WizardStyle=modern
ArchitecturesInstallIn64BitMode=x64
PrivilegesRequired=lowest
PrivilegesRequiredOverridesAllowed=dialog

[Languages]
Name: "english"; MessagesFile: "compiler:Default.isl"

[Tasks]
Name: "desktopicon"; Description: "{cm:CreateDesktopIcon}"; GroupDescription: "{cm:AdditionalIcons}"; Flags: unchecked

[Files]
; Copy the ENTIRE unpacked Electron folder produced by electron-packager
Source: "{#SourceDir}\*"; DestDir: "{app}"; Flags: ignoreversion recursesubdirs createallsubdirs

[Icons]
Name: "{group}\{#MyAppName}"; Filename: "{app}\{#MyAppExeName}"
Name: "{group}\Uninstall {#MyAppName}"; Filename: "{uninstallexe}"
Name: "{autodesktop}\{#MyAppName}"; Filename: "{app}\{#MyAppExeName}"; Tasks: desktopicon

[Run]
; Optionally launch the app right after install
Filename: "{app}\{#MyAppExeName}"; Description: "{cm:LaunchProgram,{#MyAppName}}"; Flags: nowait postinstall skipifsilent
```

**What this script does:**

- `[Setup]` block — metadata, install directory, output location, and the branded installer icon.
- `{#SourceDir}` points to the folder produced by `electron-packager`.
- `[Files]` recursively copies that folder into `C:\Program Files\Notes App\`.
- `[Icons]` creates Start Menu and (optional) Desktop shortcuts.
- `[Tasks]` adds a checkbox in the wizard for the desktop icon.
- `[Run]` lets the user launch the app immediately after install.
- The uninstaller is generated automatically and registered with Windows "Apps & Features".

#### Step 4 — Compile the installer

**From the Inno Setup IDE:** open `NotesApp.iss` and press **F9** (Compile). The finished installer appears in `release/installer/`.

**From the command line:**

```bash
"C:\Program Files (x86)\Inno Setup 6\ISCC.exe" installer\NotesApp.iss
```

Output:

```
release/
└── installer/
    └── NotesApp-Setup-1.0.0.exe     ← ship this to your users
```

Double-click it and you get a polished Windows installer wizard with your branding.

#### Step 5 — Automate the whole pipeline

Chain everything together with a single npm script in `package.json`:

```json
{
  "scripts": {
    "build:renderer": "cd renderer && npm run build",
    "build:electron": "tsc -p tsconfig.json",
    "build": "npm run build:renderer && npm run build:electron",
    "pack:win": "npm run build && electron-packager . NotesApp --platform=win32 --arch=x64 --out=release --overwrite --icon=assets/icon.ico --ignore=\"^/(renderer/src|renderer/node_modules|electron)$\"",
    "installer:win": "npm run pack:win && \"C:\\Program Files (x86)\\Inno Setup 6\\ISCC.exe\" installer\\NotesApp.iss"
  }
}
```

Now a single command produces the final branded installer:

```bash
npm run installer:win
```

### 12.3 Common Inno Setup customizations

Students will often ask for these tweaks — here are the one-liners:

**Show a license agreement**

```iss
[Setup]
LicenseFile=license.txt
```

**Show a custom README after install**

```iss
[Setup]
InfoAfterFile=readme.txt
```

**Require Windows 10 or newer**

```iss
[Setup]
MinVersion=10.0.17763
```

**Install for current user only (no admin needed)**

```iss
[Setup]
PrivilegesRequired=lowest
DefaultDirName={userpf}\{#MyAppName}
```

**Associate a file extension (e.g. `.note`) with the app**

```iss
[Registry]
Root: HKCR; Subkey: ".note"; ValueType: string; ValueName: ""; ValueData: "NotesApp.Document"; Flags: uninsdeletevalue
Root: HKCR; Subkey: "NotesApp.Document"; ValueType: string; ValueName: ""; ValueData: "Notes App Document"; Flags: uninsdeletekey
Root: HKCR; Subkey: "NotesApp.Document\DefaultIcon"; ValueType: string; ValueName: ""; ValueData: "{app}\{#MyAppExeName},0"
Root: HKCR; Subkey: "NotesApp.Document\shell\open\command"; ValueType: string; ValueName: ""; ValueData: """{app}\{#MyAppExeName}"" ""%1"""
```

**Auto-start on Windows login**

```iss
[Registry]
Root: HKCU; Subkey: "Software\Microsoft\Windows\CurrentVersion\Run"; ValueType: string; ValueName: "NotesApp"; ValueData: """{app}\{#MyAppExeName}"""; Flags: uninsdeletevalue
```

### 12.4 Code signing (optional but important in production)

Unsigned installers trigger a scary "Windows protected your PC" SmartScreen warning. To remove it you need a **code signing certificate** from a trusted CA (DigiCert, Sectigo, etc.) and the `signtool` utility from the Windows SDK.

Sign the final installer:

```bash
signtool sign /f cert.pfx /p yourpassword /tr http://timestamp.digicert.com /td sha256 /fd sha256 NotesApp-Setup-1.0.0.exe
```

Inno Setup can also sign automatically via the `SignTool` directive in the `.iss` file. For a classroom exercise you can skip signing — just explain to students it's the last step before real distribution.

## 13. Best Practices & Security

Electron apps face web-like threats *plus* desktop-like power. Teach students these rules early:

1. **Always** set `contextIsolation: true` and `nodeIntegration: false`.
2. **Never** expose `ipcRenderer` directly to the renderer. Expose a small, named, typed API through `contextBridge`.
3. **Validate** every piece of data received through IPC in the main process — TypeScript types are compile-time only; they don't protect against malicious runtime input.
4. **Don't** load remote content into the main window. If you must, use a separate `<webview>` with strict options.
5. Keep Electron **up-to-date** to get the latest Chromium security patches.
6. Use a **Content Security Policy** in your HTML:
   ```html
   <meta http-equiv="Content-Security-Policy" content="default-src 'self'">
   ```
7. Structure code so OS-touching logic lives in the main process and UI logic in React — don't mix them.
8. **Enable `"strict": true`** in every `tsconfig.json`. It catches the vast majority of student bugs.

## Teaching Tips

- **Live code every example.** Students learn ten times more watching mistakes get fixed than reading finished code.
- **Explain the main/renderer split on day one** and revisit it every lesson. It is the #1 thing students get confused by.
- **Lean on TypeScript errors.** When a student's code breaks, the red squiggles almost always tell them exactly what's wrong. Train them to read the error message first.
- **Always keep DevTools open.** Press `Ctrl+Shift+I` (or `Cmd+Opt+I` on macOS) in a running Electron window.
- **Version-control each lesson.** Commit at the end of every lesson so students can rewind.
- **Pair programming works well** — one student drives the main process (`electron/*.ts`), another the React side (`renderer/src/**/*.tsx`).
- **Treat `IElectronAPI` as sacred.** Any new feature that crosses the IPC boundary must first add a method to this interface. This keeps the contract visible and teaches good API design.

Happy teaching!
