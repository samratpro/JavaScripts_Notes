## 1. Why Next.js for SEO

A plain React app (created with Vite or Create React App) renders in the browser. When Google's crawler loads the page, it first sees an empty `<div id="root">` — the content appears only after JavaScript runs. This hurts SEO.

Next.js solves this by rendering pages **on the server**. The crawler receives a fully-populated HTML document on the first byte. The result:

| Approach | What crawlers see | SEO | Initial load |
|---|---|---|---|
| Plain React (CSR only) | Empty HTML shell | Weak | Slow until JS runs |
| Next.js (SSR-first) | Fully-rendered HTML | Excellent | Instant content |

Next.js 14's **App Router** takes this further: every component is a **Server Component** by default. You only opt into client-side rendering when you truly need browser APIs (state, effects, event handlers).

## 2. SSR vs CSR vs SSG vs ISR — The Mental Model

Teach students these four rendering modes early:

| Mode | When HTML is built | Use case |
|---|---|---|
| **SSR** (Server-Side Rendering) | On every request | Personalized pages, frequently-changing data |
| **SSG** (Static Site Generation) | At build time | Marketing pages, docs, rarely-changing content |
| **ISR** (Incremental Static Regeneration) | At build + periodically revalidated | Blog posts, product pages |
| **CSR** (Client-Side Rendering) | In the browser after JS loads | Interactive widgets inside an SSR page |

**Our philosophy for this course:** SSR (or SSG/ISR) by default. CSR only for the small parts of the page that truly need it — a menu toggle, a search box, a form input.

In Next.js App Router:

- Server Components → SSR/SSG/ISR (the default)
- Client Components → CSR (marked with `"use client"` at the top)

## 5. The Golden Rule: Server Components by Default

**Every file in `app/` is a Server Component unless you explicitly mark it as a Client Component.**

### Server Component (default, no directive)

```tsx
// src/app/page.tsx
export default async function HomePage() {
  // Runs on the server. You can await data, read files, query DBs.
  const posts = await fetch('https://api.example.com/posts').then((r) => r.json());

  return (
    <main>
      <h1>Latest Posts</h1>
      <ul>
        {posts.map((p: { id: string; title: string }) => (
          <li key={p.id}>{p.title}</li>
        ))}
      </ul>
    </main>
  );
}
```

Notice: no `useState`, no `useEffect`, no `onClick`. This renders on the server and sends pure HTML.

### Client Component (opt-in, for interactivity)

```tsx
// src/app/components/SearchBox.tsx
'use client';

import { useState } from 'react';

export default function SearchBox() {
  const [query, setQuery] = useState('');
  return (
    <input
      value={query}
      onChange={(e) => setQuery(e.target.value)}
      placeholder="Search..."
    />
  );
}
```

The `'use client'` directive says: "this component needs to run in the browser."

### How to decide

| Need | Component type |
|---|---|
| Fetching data from a DB or API | Server |
| Reading cookies or headers on the server | Server |
| Using `useState`, `useEffect`, `useContext` | Client |
| `onClick`, `onChange`, `onSubmit` | Client |
| Browser APIs (`window`, `localStorage`) | Client |
| Rendering a mostly-static page | Server |

**Teach this:** Keep Server Components as the outer shell and push Client Components as deep and small as possible. A small `<LikeButton />` inside an otherwise-server-rendered blog post is the ideal pattern.

## 6. SEO Essentials: Metadata, Open Graph, Twitter Cards

Next.js has a built-in **Metadata API**. You export a `metadata` object (static) or a `generateMetadata` function (dynamic) from any `layout.tsx` or `page.tsx`.

### Static metadata (site-wide defaults in root layout)

```tsx
// src/app/layout.tsx
import type { Metadata } from 'next';
import './globals.css';

export const metadata: Metadata = {
  metadataBase: new URL('https://example.com'),
  title: {
    default: 'My SEO App',
    template: '%s | My SEO App',        // e.g. "Blog | My SEO App"
  },
  description: 'A blazing-fast SSR site built with Next.js and TypeScript.',
  keywords: ['Next.js', 'React', 'SEO', 'SSR', 'TypeScript'],
  authors: [{ name: 'Samrat', url: 'https://example.com' }],
  creator: 'Samrat',
  publisher: 'Samrat',
  robots: {
    index: true,
    follow: true,
    googleBot: {
      index: true,
      follow: true,
      'max-image-preview': 'large',
      'max-snippet': -1,
    },
  },
  openGraph: {
    type: 'website',
    locale: 'en_US',
    url: 'https://example.com',
    siteName: 'My SEO App',
    title: 'My SEO App',
    description: 'A blazing-fast SSR site built with Next.js.',
    images: [
      {
        url: '/og-image.png',
        width: 1200,
        height: 630,
        alt: 'My SEO App',
      },
    ],
  },
  twitter: {
    card: 'summary_large_image',
    site: '@yoursite',
    creator: '@yourhandle',
    title: 'My SEO App',
    description: 'A blazing-fast SSR site built with Next.js.',
    images: ['/og-image.png'],
  },
  alternates: {
    canonical: 'https://example.com',
    languages: {
      'en-US': '/en-US',
      'hi-IN': '/hi-IN',
    },
  },
  verification: {
    google: 'your-google-site-verification-code',
  },
};

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}
```

Next.js injects all the correct `<title>`, `<meta>`, `<link rel="canonical">`, and OpenGraph tags into the `<head>`.

### Dynamic metadata (per-page, data-driven)

For blog posts where the title, description, and OG image depend on the post:

```tsx
// src/app/blog/[slug]/page.tsx
import type { Metadata } from 'next';
import { getPost } from '@/lib/posts';

interface PageProps {
  params: { slug: string };
}

export async function generateMetadata({ params }: PageProps): Promise<Metadata> {
  const post = await getPost(params.slug);

  return {
    title: post.title,
    description: post.excerpt,
    openGraph: {
      title: post.title,
      description: post.excerpt,
      type: 'article',
      publishedTime: post.publishedAt,
      authors: [post.author],
      images: [{ url: post.coverImage, width: 1200, height: 630 }],
    },
    twitter: {
      card: 'summary_large_image',
      title: post.title,
      description: post.excerpt,
      images: [post.coverImage],
    },
    alternates: {
      canonical: `https://example.com/blog/${post.slug}`,
    },
  };
}

export default async function BlogPostPage({ params }: PageProps) {
  const post = await getPost(params.slug);
  return (
    <article>
      <h1>{post.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.html }} />
    </article>
  );
}
```

## 7. Favicon, Logo, and Icons

Next.js App Router has a convention-based system for icons. Place these files directly inside `src/app/`:

| File in `src/app/` | Result in HTML |
|---|---|
| `favicon.ico` | `<link rel="icon">` |
| `icon.png` or `icon.svg` | `<link rel="icon">` (modern) |
| `apple-icon.png` | `<link rel="apple-touch-icon">` |
| `opengraph-image.png` | Used as default OG image |
| `twitter-image.png` | Used as default Twitter card image |

### Recommended set of files

```
src/app/
├── favicon.ico                    # 32x32 or 48x48
├── icon.png                       # 512x512 modern icon
├── apple-icon.png                 # 180x180
├── opengraph-image.png            # 1200x630 — default social preview
└── twitter-image.png              # 1200x630 — default twitter card
```

### Programmatic icon generation (dynamic)

You can even generate an icon with code:

```tsx
// src/app/icon.tsx
import { ImageResponse } from 'next/og';

export const size = { width: 32, height: 32 };
export const contentType = 'image/png';

export default function Icon() {
  return new ImageResponse(
    (
      <div
        style={{
          fontSize: 24,
          background: '#000',
          color: 'white',
          width: '100%',
          height: '100%',
          display: 'flex',
          alignItems: 'center',
          justifyContent: 'center',
          borderRadius: '50%',
        }}
      >
        M
      </div>
    ),
    { ...size }
  );
}
```

### Logo in the header (static asset)

Put the logo in `public/` and load it with the optimized `<Image>` component:

```tsx
// src/app/components/Header.tsx
import Image from 'next/image';
import Link from 'next/link';

export default function Header() {
  return (
    <header className="flex items-center gap-4 p-4 border-b">
      <Link href="/" className="flex items-center gap-2">
        <Image
          src="/logo.svg"
          alt="My SEO App logo"
          width={40}
          height={40}
          priority
        />
        <span className="font-bold text-xl">My SEO App</span>
      </Link>
    </header>
  );
}
```

`next/image` automatically optimizes, lazy-loads, and serves the best size/format (WebP/AVIF) for each device.

## 8. Sitemap.xml Generation

Create `src/app/sitemap.ts` — Next.js will serve it at `/sitemap.xml`:

```typescript
// src/app/sitemap.ts
import type { MetadataRoute } from 'next';
import { getAllPosts } from '@/lib/posts';

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const baseUrl = 'https://example.com';

  // Static routes
  const staticRoutes: MetadataRoute.Sitemap = [
    {
      url: baseUrl,
      lastModified: new Date(),
      changeFrequency: 'daily',
      priority: 1.0,
    },
    {
      url: `${baseUrl}/about`,
      lastModified: new Date(),
      changeFrequency: 'monthly',
      priority: 0.7,
    },
    {
      url: `${baseUrl}/blog`,
      lastModified: new Date(),
      changeFrequency: 'daily',
      priority: 0.9,
    },
    {
      url: `${baseUrl}/contact`,
      lastModified: new Date(),
      changeFrequency: 'yearly',
      priority: 0.5,
    },
  ];

  // Dynamic routes — one entry per blog post
  const posts = await getAllPosts();
  const postRoutes: MetadataRoute.Sitemap = posts.map((post) => ({
    url: `${baseUrl}/blog/${post.slug}`,
    lastModified: new Date(post.updatedAt),
    changeFrequency: 'weekly',
    priority: 0.8,
  }));

  return [...staticRoutes, ...postRoutes];
}
```

Visit [http://localhost:3000/sitemap.xml](http://localhost:3000/sitemap.xml) — you'll see a valid sitemap.

## 9. Robots.txt Generation

Create `src/app/robots.ts` — served at `/robots.txt`:

```typescript
// src/app/robots.ts
import type { MetadataRoute } from 'next';

export default function robots(): MetadataRoute.Robots {
  return {
    rules: [
      {
        userAgent: '*',
        allow: '/',
        disallow: ['/admin/', '/api/', '/dashboard/', '/private/'],
      },
      {
        userAgent: 'Googlebot',
        allow: '/',
        disallow: ['/admin/', '/api/'],
      },
    ],
    sitemap: 'https://example.com/sitemap.xml',
    host: 'https://example.com',
  };
}
```

## 10. Structured Data (JSON-LD)

Structured data helps Google show rich results (star ratings, article cards, breadcrumbs). Inject a JSON-LD `<script>` from a Server Component:

```tsx
// src/app/blog/[slug]/page.tsx
export default async function BlogPostPage({ params }: { params: { slug: string } }) {
  const post = await getPost(params.slug);

  const jsonLd = {
    '@context': 'https://schema.org',
    '@type': 'BlogPosting',
    headline: post.title,
    description: post.excerpt,
    image: post.coverImage,
    datePublished: post.publishedAt,
    dateModified: post.updatedAt,
    author: {
      '@type': 'Person',
      name: post.author,
    },
    publisher: {
      '@type': 'Organization',
      name: 'My SEO App',
      logo: { '@type': 'ImageObject', url: 'https://example.com/logo.png' },
    },
  };

  return (
    <>
      <script
        type="application/ld+json"
        dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
      />
      <article>
        <h1>{post.title}</h1>
        <div dangerouslySetInnerHTML={{ __html: post.html }} />
      </article>
    </>
  );
}
```

Because this is a Server Component, the JSON-LD is in the initial HTML — perfect for crawlers.

### `src/lib/posts.ts` (file-system-based data)

```typescript
import fs from 'fs/promises';
import path from 'path';
import matter from 'gray-matter';
import { marked } from 'marked';
import type { Post } from '@/types/post';

const POSTS_DIR = path.join(process.cwd(), 'src/content/posts');

export async function getAllPosts(): Promise<Post[]> {
  const files = await fs.readdir(POSTS_DIR);
  const posts = await Promise.all(
    files
      .filter((f) => f.endsWith('.md'))
      .map(async (filename) => {
        const slug = filename.replace(/\.md$/, '');
        return getPost(slug);
      })
  );
  return posts.sort((a, b) => (a.publishedAt < b.publishedAt ? 1 : -1));
}

export async function getPost(slug: string): Promise<Post> {
  const file = await fs.readFile(path.join(POSTS_DIR, `${slug}.md`), 'utf-8');
  const { data, content } = matter(file);
  const html = await marked.parse(content);

  return {
    slug,
    title: data.title,
    excerpt: data.excerpt,
    author: data.author,
    publishedAt: data.publishedAt,
    updatedAt: data.updatedAt ?? data.publishedAt,
    coverImage: data.coverImage,
    category: data.category,
    html,
  };
}
```

```tsx
export async function generateStaticParams() {

```

#### `src/middleware.ts` — auth-gate the dashboard

```typescript
import { NextRequest, NextResponse } from 'next/server';

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;
  const isAuth = Boolean(request.cookies.get('session-token'));

  // Protect every /dashboard/* route
  if (pathname.startsWith('/dashboard') && !isAuth) {
    const url = request.nextUrl.clone();
    url.pathname = '/login';
    url.searchParams.set('callbackUrl', pathname);
    return NextResponse.redirect(url);
  }

  return NextResponse.next();
}

export const config = {
  matcher: ['/dashboard/:path*', '/users/:path*', '/products/:path*', '/settings/:path*'],
};
```

## 13. Data Fetching Patterns (SSR-first)

Next.js extends `fetch` with caching hints that control rendering mode:

```typescript
// SSR: fetch on every request
const res = await fetch('https://api.example.com/data', { cache: 'no-store' });

// SSG: fetch once at build time (default when no options)
const res = await fetch('https://api.example.com/data');

// ISR: fetch and cache, revalidate every N seconds
const res = await fetch('https://api.example.com/data', { next: { revalidate: 60 } });

// Tag-based revalidation (call revalidateTag('posts') to bust this)
const res = await fetch('https://api.example.com/posts', { next: { tags: ['posts'] } });
```

### Server Actions (forms without CSR)

Server Actions let students write forms that submit without any client-side JavaScript:

```tsx
// src/app/contact/page.tsx
export default function ContactPage() {
  async function submit(formData: FormData) {
    'use server';
    const name = formData.get('name') as string;
    const email = formData.get('email') as string;
    // save to DB directly from the server
    await db.contact.create({ data: { name, email } });
  }

  return (
    <form action={submit}>
      <input name="name" required />
      <input name="email" type="email" required />
      <button type="submit">Send</button>
    </form>
  );
}
```

This form works **with JavaScript disabled** — the ultimate SSR-friendly form.

## 14. Performance & Core Web Vitals

SEO rankings depend heavily on Core Web Vitals. Teach students these habits:

1. **Use `next/image`** for every image — automatic optimization, lazy-loading, WebP/AVIF.
2. **Use `next/font`** for web fonts — zero layout shift, preloaded:
   ```tsx
   import { Inter } from 'next/font/google';
   const inter = Inter({ subsets: ['latin'] });
   <html lang="en" className={inter.className}>...
   ```
3. **Minimize Client Components** — push `'use client'` as deep and small as possible.
4. **Stream with `loading.tsx`** — show a skeleton while data loads:
   ```tsx
   // src/app/blog/loading.tsx
   export default function Loading() {
     return <div className="animate-pulse">Loading posts...</div>;
   }
   ```
5. **Cache aggressively** with `revalidate` and tag-based invalidation.
6. **Preload critical links** with `<Link prefetch>` (default in Next.js).
7. **Audit with Lighthouse** — press F12 in Chrome → Lighthouse tab → Analyze page load.

## Teaching Tips

- **Start by viewing page source.** Open a plain Create-React-App site and a Next.js site side by side, then hit Ctrl+U on each. The empty `<div id="root">` vs. fully-rendered HTML is the single most convincing demo of why Next.js exists.
- **Make "`'use client'` is the exception, not the rule" a mantra.** Every time a student reaches for state or effects, ask "could this be server-rendered instead?"
- **Teach the Metadata API before teaching components.** SEO is often ignored in tutorials; by front-loading it you build the right habits from day one.
- **Use the Lighthouse tab religiously.** Students should see their score change in real time as they add images without `next/image`, add client components needlessly, or skip font optimization.
- **Treat `sitemap.ts` and `robots.ts` as production requirements** — not optional. Don't let a student call a project "done" without both.
- **Version-control each lesson** and deploy each lesson to Vercel preview URLs so students can see their work publicly.

Happy teaching!
